执行摘要与关键修订

SnapEdit（闪书）是 MMY Markdown 工具矩阵的"写"端，与 SnapMD（闪阅）构成完整工作流闭环。本版在 v0.2 基础上新增公众号格式导出和双向链接知识网络，将产品从纯编辑器升级为轻量知识书写工具。

版本变更影响
v0.2 内核 Tiptap → CodeMirror 6；Editing Contract；MVP 4 → 6 周；Phase 0 量化门槛；测试策略 修正根本性架构风险，保证源文本保真
v0.3 公众号格式导出（Phase 1 W5 同周）+ 双向链接知识网络（新增 Phase 1.5） 两项高杠杆刚需功能，3.5 天 + 1.5 周增量，产品定位升级

产品定位演进

v0.1: Markdown 编辑器（写 .md 文件）

v0.2: 极简高效 Markdown 编辑器（源文本保真 + 编辑契约）

v0.3: 轻量知识书写工具
      公众号作者：写 MD → 一键导出微信格式
      知识管理：写 MD → [[链接]] 编织知识网络
2

编辑内核选型：CodeMirror 6

原方案最大风险：Tiptap 的 HTML ↔ Markdown 双向转换失真

Tiptap 内部存 JSON，Markdown 只是序列化格式。每次打开/切换模式都经过转换，可能丢失格式。

CodeMirror 6（主推）

数据模型纯文本 + Decoration 渲染层
源文本保真★★★★★ 编辑即原文
WYSIWYGDecoration 叠加（标题放大、粗体变粗）
大文件★★★★★ 原生虚拟滚动
拓展性CM6 Extension + [[wiki-link]] 自定义

Tiptap（仅对照 Spike）

数据模型ProseMirror JSON
源文本保真★★ 双向转换可能丢失
WYSIWYG★★★★★ 原生真 WYSIWYG
大文件★★★ 需分块渲染
拓展性Extension API 丰富

编辑契约 (Editing Contract)

Round-trip 保真

打开 → 不修改 → 保存，逐字节一致

不引入不可见字符

零 HTML 标签、零宽字符、BOM

编码保持

GBK 文件保存仍是 GBK

空白不改变

缩进和行尾符号保持一致

大文件策略

>500KB 确认，>2MB 只读

降级兜底

编码失败保留原始字节

公众号格式导出 NEW v0.3

用 3.5 天开发量换一个几乎没有竞品做好的刚需场景：写 MD → 一键得到微信可粘贴的富文本。

微信限制对策
只认内联样式（class 全被过滤）juice 库将 CSS class → 内联 style=""
代码块无高亮且极丑渲染为 Carbon 风格 SVG 或内联高亮 <pre>
图片被阻止v1.0 base64 内嵌（<32KB），v1.1 对接图床
表格撑破 375px 宽度外包 overflow-x:auto + 缩小字号
链接不可点击正文保留格式，末尾追加脚注展示 URL

导出数据流

MD 源文本 → markdown-it → HTML → juice 内联 CSS → 微信后处理 → 375px 预览 → 复制

排期（Phase 1 W5 导出周，3.5 天）

子任务工时
juice 内联化实现0.5 天
微信专用 CSS 模板1 天
代码块 SVG 渲染1 天
手机宽度预览窗 + 复制1 天

双向链接与知识网络 NEW v0.3

引入工作区概念和 [[wiki-link]] 语法，将 SnapEdit 从"单文件编辑器"升级为"轻量知识书写工具"。

UI 交互设计

[[ 触发自动补全菜单，模糊匹配所有 .md
[[file]] 高亮已存在 → 蓝色下划线；不存在 → 橙色虚线
Ctrl+Click保存当前 → 跳转到目标文件
鼠标悬浮显示目标文件标题 + 前 3 行
[[新笔记]]自动创建同名 .md 文件
反向链接面板侧边栏展示引用关系 + 上下文片段

数据模型

工作区用户打开的文件夹，含 N 个 .md
索引构建启动时 Rust 异步扫描，不阻塞 UI
增量更新保存时只扫当前文件的 outgoing links
outgoing本文件引用了谁
backlinks谁引用了本文件（反向计算）
