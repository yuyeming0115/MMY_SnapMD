# SnapMD 闪阅 —— 多格式阅读与编辑功能迭代计划

> 制定日期：2026-07-07
> 适用阶段：SnapMD v0.2 功能扩展规划
> 目标：在保留“快速阅读”的前提下，增加 `.json`、`.txt` 正式支持与轻量编辑能力，并重排主界面信息结构。

---

## 一、结论

本轮迭代建议把 SnapMD 从“Markdown 预览器”升级为“轻量文本类文档阅读器”，但不要直接做成完整 IDE 或 SnapEdit。

推荐路线：

1. **先抽象文档类型**：把现有 `Markdown` 命名改为通用 `Document` / `TextDocument`，支持 `markdown`、`txt`、`json` 三类。
2. **先做轻量编辑**：源文本编辑 + 预览同步 + 手动保存 + 脏状态提示；暂不做复杂富文本、自动格式化保存、批量替换。
3. **先重排 UI 骨架**：顶部水平工具条承载打开、保存、编辑、主题、导出；左侧优先放目录/结构，文件信息下沉到目录底部。
4. **JSON 先做可读预览**：格式化代码视图 + 结构目录；编辑时校验 JSON，允许保存原文但明确提示语法状态。
5. **TXT 走纯文本阅读模式**：保留换行、等宽排版、行号/行数统计；不再强行当 Markdown 渲染。

---

## 二、当前状态判断

现有代码已经具备这些基础：

- 前端：`Vue 3 + TypeScript`，主逻辑集中在 `MMY_SnapMD/src/App.vue`。
- 渲染：`MMY_SnapMD/src/markdown.ts` 使用 `markdown-it + highlight.js + DOMPurify`。
- 文件入口：支持文件输入、浏览器拖拽、Tauri 桌面拖拽、启动参数打开。
- 文件类型：当前实际支持 `.md` / `.markdown` / `.txt`，但命名、提示、Tauri 命令仍以 `markdown` 为中心。
- 自动刷新：Tauri 桌面模式按文件修改时间轮询，外部修改后自动重读。
- UI：左侧依次为品牌、打开按钮、功能按钮、最近打开、文件信息、目录；阅读区域为居中纸张式预览。

主要问题：

- `.txt` 虽然可打开，但仍走 Markdown 渲染链，产品语义不清。
- `.json` 未进入文件过滤、后端白名单、文件关联与渲染链。
- 编辑功能与自动刷新存在冲突：用户正在编辑时，外部文件变化不能直接覆盖当前内容。
- 功能按钮放在左侧会挤占“目录”这个高频区域。
- 文件信息比目录低频，当前位置偏高。

## 三、产品范围

### 3.1 本轮进入范围

| 能力 | 范围 | 优先级 |
|------|------|--------|
| Markdown 阅读 | 维持现有渲染能力 | P0 |
| TXT 阅读 | 纯文本预览、保留空白、统计行数/字符数 | P0 |
| JSON 阅读 | 格式化预览、代码高亮、结构目录、语法错误提示 | P0 |
| 轻量编辑 | 编辑/预览切换、内容修改、保存、另存为后置 | P0/P1 |
| 顶部工具条 | 打开、保存、编辑开关、主题、导出、同步状态 | P0 |
| 左侧目录优先 | Markdown 标题目录 / JSON 结构目录 / TXT 可降级为空状态 | P0 |
| 文件信息下沉 | 放到左侧底部或可折叠区域 | P0 |
| 最近打开 | 保留，但高度受限，不抢占目录 | P1 |

### 3.2 本轮暂不进入范围

- 多标签页
- 文件夹工作区
- 全文搜索与替换
- JSON 可视化表单编辑
- 自动保存默认开启
- Git diff / 版本历史
- 富文本编辑、所见即所得编辑
- 复杂编码检测与 GBK 保真保存

## 四、UI 迭代方案

### 4.1 推荐布局

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部工具条：Logo / 打开 / 保存 / 编辑 / 预览模式 / 导出 / 主题 / 状态 │
├───────────────┬─────────────────────────────────────────────┤
│ 左侧栏         │ 主内容区                                      │
│               │                                             │
│ 目录/结构      │ 阅读模式：Markdown / TXT / JSON 预览           │
│ - Markdown H1 │ 编辑模式：源文本编辑器                         │
│ - JSON key    │ 分屏模式：左编辑 / 右预览（P1）                 │
│               │                                             │
│ 最近打开(P1)   │                                             │
│               │                                             │
│ 文件信息       │                                             │
└───────────────┴─────────────────────────────────────────────┘
```

### 4.2 顶部工具条内容

| 区域 | 控件 | 说明 |
|------|------|------|
| 左侧 | Logo + 当前文件名 | 品牌保留，但减少高度占用 |
| 文件操作 | 打开、保存、另存为（P1） | 保存按钮只在编辑或脏状态下高亮 |
| 模式操作 | 阅读 / 编辑 / 分屏（P1） | MVP 可先做阅读/编辑两态 |
| 导出操作 | PDF、HTML | Markdown 沿用现有能力；TXT/JSON 导出 HTML 可后置 |
| 视图操作 | 主题、字体大小（P1） | 主题保留；字体大小适合阅读器 |
| 状态 | 已保存 / 未保存 / 监听中 / 外部已变更 | 替代原侧栏同步字段 |

### 4.3 左侧栏重排

左侧栏建议改为“三段式”：

1. **目录/结构导航（主区域）**
   - Markdown：沿用标题目录。
   - JSON：显示对象 key / 数组节点结构，默认展开 2 层。
   - TXT：没有结构时显示“当前文件无目录”，可后续增加行号跳转。

2. **最近打开（次区域）**
   - 放在目录下方，最大高度受限。
   - 用户若常用目录，最近打开不应挤占目录滚动空间。

3. **文件信息（底部区域）**
   - 文件路径、类型、行数、字符数、打开时间、同步状态。
   - 可做成紧凑信息块或折叠面板。

### 4.4 主内容区模式

| 模式 | 说明 | MVP |
|------|------|-----|
| 阅读模式 | 当前主预览体验，按文档类型选择渲染器 | 必做 |
| 编辑模式 | 单栏源文本编辑，预览隐藏 | 必做 |
| 分屏模式 | 左源文本、右预览，适合 Markdown / JSON | P1 |

## 五、技术设计

### 5.1 文档类型抽象

新增统一类型：

```ts
type DocumentKind = 'markdown' | 'txt' | 'json';

type OpenedDocument = {
  path: string;
  name: string;
  content: string;
  kind: DocumentKind;
  modifiedAtMs?: number | null;
};
```

文件识别规则：

| 后缀 | kind | 渲染器 |
|------|------|--------|
| `.md` / `.markdown` | `markdown` | `renderMarkdown` |
| `.txt` | `txt` | `renderPlainText` |
| `.json` | `json` | `renderJson` |

### 5.2 渲染器拆分

建议新增 `src/document.ts` 或 `src/renderers/`：

```text
src/
├── document.ts              # 文档类型、后缀识别、统计
├── markdown.ts              # 现有 Markdown 渲染链
├── renderers/
│   ├── plainText.ts         # TXT 转义 + <pre>
│   └── json.ts              # JSON parse/format/highlight/structure
```

统一渲染结果：

```ts
type RenderResult = {
  html: string;
  headings: Array<{ id: string; level: number; text: string }>;
  diagnostics?: Array<{ type: 'error' | 'warning'; message: string }>;
};
```

### 5.3 JSON 渲染策略

MVP 不做复杂树控件，先做两件事：

1. **格式化代码预览**
   - `JSON.parse(content)` 成功：`JSON.stringify(value, null, 2)` 后用 highlight.js 的 `json` 高亮。
   - 失败：显示原文代码块 + 错误提示，不阻断编辑。

2. **结构目录**
   - 对对象 key 和数组节点生成 heading。
   - 默认提取深度 1-3 层，避免大 JSON 目录过载。
   - 长 key 截断展示，title 保留完整路径。

### 5.4 TXT 渲染策略

- 使用 `<pre>` 或专用 `.plain-text-body` 保留换行和空格。
- 默认不开启 Markdown 语义解析，避免普通文本里的 `#`、`*` 被误渲染。
- 统计从“词数”调整为更通用的“字符数 / 行数”；Markdown 可继续显示词数。

### 5.5 编辑与保存策略

编辑功能的安全底线：

| 规则 | 说明 |
|------|------|
| 源文本是唯一真相 | 编辑区直接修改 `source`，预览由 `source` 派生 |
| 默认手动保存 | MVP 不默认自动保存，降低误写风险 |
| 脏状态明确 | 修改后顶部显示“未保存”，关闭/切换文件前确认 |
| 外部变更不覆盖编辑 | 编辑且未保存时，文件监听只提示“外部已变更”，不自动重读 |
| 保存后更新修改时间 | 调用后端保存成功后刷新 `modifiedAtMs` |
| JSON 语法错误可保存 | 这是文本编辑器，不强制阻止保存，但保存前提示 |

Tauri 后端建议新增命令：

```rust
read_document_file(path)
document_file_state(path)
save_document_file(path, content)
```

并把 `ensure_supported_markdown` 改为 `ensure_supported_document`，支持：

```text
md, markdown, txt, json
```

## 七、优先级建议

建议先做：

1. UI 骨架重排
2. 通用文档类型抽象
3. `.json` / `.txt` 正式阅读
4. 手动编辑保存

不建议一开始就做：

1. CodeMirror 6 完整接入
2. 自动保存
3. 多标签
4. JSON 树编辑器

原因：SnapMD 当前价值仍是“打开快、阅读稳”。编辑能力应先补“必要修改”的场景，等使用频率确认后，再决定是否向 SnapEdit 靠拢。

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 编辑状态被自动刷新覆盖 | 用户修改丢失 | 未保存时禁用自动覆盖，只提示外部变更 |
| TXT 被 Markdown 误渲染 | 阅读体验不可信 | TXT 单独渲染为纯文本 |
| JSON 大文件解析卡顿 | UI 卡顿 | 目录提取限制深度和节点数；大文件降级为代码预览 |
| 保存接口误写不支持格式 | 数据风险 | 后端统一白名单；保存前复用同一校验 |
| UI 顶部工具条过满 | 小窗口拥挤 | 次要功能放入“更多”菜单或后置到 P1 |
| 与 SnapEdit 产品边界重叠 | 定位模糊 | SnapMD 只做轻量源文本编辑，不做完整写作工作区 |

## 十一、下一阶段建议（2026-07-07）

结论：有，而且下一步建议继续做“轻量多格式文档工具”的体验补强，不急着扩到 Word / Excel / 富文本那条重路线。

建议按下面 4 项往前推：

1. **保存与冲突提示补强（P0）**
   - 把“未保存”“外部已变更”“保存失败”“可重新载入”做成更清晰的顶部状态条或轻提示。
   - 增加“比较保守”的冲突处理动作，例如：继续编辑、放弃本地修改并重载。
   - 目标：先把“不会误覆盖”这件事做扎实。

## 十三、本轮问题经验总结（2026-07-07）

1. **桌面端拖拽不能只按“文件类型”判断，还要按“落点位置”判断**
   - Tauri 的全局拖拽事件会优先拿到文件路径，如果前端只判断“是不是支持的文件”，图片拖入编辑区时就会被误走到“打开文档”逻辑。
   - 正确做法是同时利用 `event.payload.position` 和文件路径类型：落在编辑区的图片走“插入图片”，其他区域的文档再走“打开文件”。

2. **桌面端图片拖拽和浏览器图片拖拽不是同一条链路**
   - 浏览器内拖拽能直接拿到 `File` / `DataTransfer`；Tauri 全局拖拽常见的是直接给路径数组。
   - 所以图片插入要分成两套实现：浏览器侧用 `File` 字节写入，桌面侧补一个“按源路径复制到 assets”的后端命令，不能硬共用一套前端逻辑。

3. **Vue 模板里的事件处理函数，最好不要直接复用“带业务参数”的方法**
   - 一旦函数签名改成 `saveCurrentDocument(options?)`，再在模板写 `@click="saveCurrentDocument"`，TypeScript 就会把它当成事件处理器签名去比对，容易报类型错误。
   - 更稳的做法是额外包一层无参函数，例如 `handleSaveButtonClick()`，模板只绑定这个入口。

4. **“稍后处理”不等于“风险已消失”**
   - 外部文件变更提示如果直接清掉 `externalChangeDetected`，后续保存会失去必要确认，风险太高。
   - 更安全的策略是：只收起提示条，不清理冲突状态；等用户重新载入或覆盖保存后再真正复位。

5. **Windows 打包要尽早验证中文名与安装器链路**
   - Tauri + WiX 在中文产品名场景下，默认 `en-US` / codepage 1252` 很容易触发打包失败。
   - 如果产品名确定保留中文，最好尽早把 WiX 语言切到 `zh-CN`，避免最后阶段才踩到安装包问题。
