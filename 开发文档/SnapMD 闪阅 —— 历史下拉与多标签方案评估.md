# SnapMD 闪阅 —— 历史下拉与多标签方案评估

## 一、结论

建议优先落地“顶部历史下拉”，暂不立即做完整浏览器式多标签。

- 历史下拉：高可行、高收益、低风险，能直接释放左侧目录空间，符合 SnapMD “打开快、阅读稳、轻量可改”的定位。
- 浏览器式多标签：需求成立，但当前性价比偏低；它会把现有单文档状态模型升级为多文档会话模型，牵动保存、外部变更监听、拖拽打开、启动参数、最近记录、分屏编辑等多处逻辑。
- 推荐路径：先做“顶部历史下拉 + 最近文件快速切换”，观察是否足够解决多文件切换痛点；再按使用频率决定是否进入轻量标签页或完整多标签。

## 二、现状依据

当前 `App.vue` 仍是单文档状态模型，核心状态包括 `source`、`savedSource`、`fileName`、`filePath`、`fileBaseDir`、`fileKind`、`viewMode`、`fileWatchTimer` 等。最近打开记录使用 `recentDocuments`，保存在 `localStorage`，当前最多保留 8 条。

当前界面中：

- 顶部工具条已经承担打开、保存、模式切换、字体、主题、导出、状态展示。
- 左侧栏主要承担目录/结构导航，同时还放置最近打开、文件信息、错误提示。
- 最近打开列表样式限制为约 150px 高度，既放不下很多记录，也会占用目录纵向空间。
- 既有开发计划曾明确把“多标签”放在后置能力，原因是当前阶段优先保证轻量阅读与必要编辑。

## 三、方案 A：顶部历史下拉

### 目标

把左侧“最近打开”从固定侧栏区域移到顶部工具条，以按钮触发下拉列表，释放左侧目录空间。

### 建议交互

- 顶部增加“最近”按钮，显示在“打开”附近。
- 点击展开历史列表，再次点击、按 Esc、点击外部区域后收起。
- 下拉列表最多展示 20 条，超过后滚动，不撑高页面。
- 每条显示文件名、格式、打开时间，路径放在 `title` 或次级小字中。
- 当前文件高亮。
- 每条保留“打开”和“移除”能力；底部保留“清空历史”。
- 打开历史文件前沿用现有未保存确认逻辑。

### 实现成本

低，约 0.5-1 天。

主要改动：

- 将 `MAX_RECENT_DOCUMENTS` 从 8 提高到 20 或 30。
- 增加 `isRecentMenuOpen` 状态与外部点击/Esc 收起逻辑。
- 移除或隐藏左侧固定 `recent-panel`，将列表模板迁移到顶部下拉。
- 调整 CSS：增加 `.recent-menu`、`.recent-dropdown`、滚动高度、窄屏布局。
- 保留原有 `loadRecentDocuments`、`saveRecentDocuments`、`openFilePath`、`removeRecentDocument`、`clearRecentDocuments` 数据逻辑。

### 性价比

最高。它直接解决“历史区域太小”和“不占用目录区域”两个明确痛点，且基本不改变文件打开、保存、监听和编辑模型。

## 四、方案 B：浏览器式水平多标签

### 目标

允许同时打开多个文件，以浏览器标签形式水平排列，点击标签切换当前文档。

### 可行性

技术上可行，但不是小 UI 改动。当前应用以“一个当前文件”为中心，多标签需要把单个文档状态升级为会话集合。

至少需要新增：

- `openDocuments` 列表或 Map，保存每个文档的 `path/name/kind/source/savedSource/baseDir/viewMode/modifiedAt/watchState`。
- `activeDocumentId` 表示当前标签。
- 每个标签独立脏状态、保存状态、外部变更状态、最后打开时间。
- 关闭标签时的未保存确认。
- 打开同一路径时复用已有标签，不重复打开。
- 拖拽、启动参数、最近打开、快捷键保存，都要定位到当前活动标签。
- 外部文件监听需要改为只监听活动标签，或为多标签建立可控的多文件监听策略。

### 实现成本

中高，约 2-4 天做轻量版，4-7 天做完整稳定版。

轻量版只做：

- 多文件标签切换。
- 每个标签保留内容和脏状态。
- 关闭标签确认未保存。
- 当前活动标签才做外部变更检测。

完整稳定版还要补：

- 启动参数打开多个文件。
- 拖拽多个文件批量打开。
- 标签溢出滚动、关闭按钮、快捷键切换。
- 标签恢复上次会话。
- 同一路径重复打开、文件被删除、外部变更冲突等边界处理。

### 性价比

当前阶段中等偏低。多标签能提升多文件阅读效率，但它会明显增加状态复杂度和测试面，也容易把 SnapMD 推向 SnapEdit 的“项目级编辑器”方向。

## 五、推荐路线

### Phase 1：先做顶部历史下拉

优先级最高。交付后左侧只保留目录/结构和文件信息，历史通过顶部下拉承担“快速切换最近文件”的任务。

落地状态：已于 2026-07-08 完成。顶部工具栏新增“最近”下拉，左侧固定历史面板已移除，最近记录上限提高到 24 条。

验收标准：

- 1280x800 下左侧目录不再被最近打开列表挤占。
- 顶部历史下拉至少可浏览 20 条记录。
- 当前文件高亮，移除和清空可用。
- 打开历史文件前仍会处理未保存修改确认。
- `.md/.txt/.json/.csv/.yaml/.yml` 最近记录都能正常打开。

### Phase 2：做轻量“打开文件栈”验证多文件需求

如果历史下拉仍不能满足多文件切换，再加入轻量文件栈，而不是马上做完整浏览器标签。

建议先做到：

- 最近打开下拉支持固定当前常用文件。
- 可选增加“当前会话文件”分组，显示本次打开过的文件。
- 仍保持一次只激活一个文档，不引入完整多标签状态。

### Phase 3：按需进入多标签

当用户明确需要在多个文件之间频繁编辑、对照、保存，且历史下拉/会话文件栈不够用时，再做多标签。

多标签进入条件建议：

- 用户经常同时处理 3 个以上文件。
- 需要保留每个文件的未保存编辑状态。
- 需要在多个文件之间来回对照，而不是偶尔重新打开。
- SnapMD 产品定位接受从“轻量预览器”向“轻量多文档编辑器”前进。

## 六、风险与取舍

- 顶部工具条可能拥挤：历史按钮应放在“打开”旁边，必要时将导出/主题等低频功能后续收进“更多”。
- 下拉层级遮挡正文：下拉最大高度限制在 360-420px，使用浮层，不改变页面布局。
- 多标签状态风险：保存、外部变更、拖拽打开和关闭确认必须按单标签隔离，否则容易出现保存错文件或覆盖外部修改。
- 产品边界风险：完整多标签会弱化 SnapMD 与 SnapEdit 的边界，建议先用历史下拉验证真实需求。
