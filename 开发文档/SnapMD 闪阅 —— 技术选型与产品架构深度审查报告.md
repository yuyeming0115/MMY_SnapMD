### 2.2 技术对比表数据需更新

**问题：**
- Tauri 2.0 的内存占用可能高于原文描述的 80-120MB（WebView2 在 Windows 上的实际表现）
- Avalonia 的 Markdown 渲染已有成熟方案（Markdown.Avalonia 组件），原文低估了可行性
- 缺少 Electron 作为对比基准（虽然不推荐，但作为行业参照有必要）

**建议补充对比维度：**
- 热更新/文件监听能力
- 插件扩展生态
- 自动更新机制
- 代码签名与分发复杂度
- 社区活跃度与维护状态

### 2.3 推荐逻辑需强化

**原文结论：** Tauri ⭐⭐⭐⭐⭐，Avalonia ⭐⭐⭐

**问题分析：**
- 如果产品定位是"极简预览器"，Avalonia 的 30-50MB 内存优势实际上非常显著
- Tauri 的 Web 技术栈在 Markdown 渲染上有天然优势，但需要评估 WebView2 在 Windows 7/旧系统上的兼容性
- 建议增加决策矩阵：根据团队技术背景（Rust/C#/全栈）给出不同推荐

**改进建议：**
```
推荐策略：
- 团队有 Rust/Web 经验 → Tauri 2.0（渲染效果最佳，开发效率高）
- 团队有 C#/.NET 经验 → Avalonia 11（内存最优，原生体验）
- 快速验证 MVP → PySide6 或 Tauri（开发速度最快）
```

## 三、架构设计缺陷与风险

### 3.1 伪代码存在严重技术问题 ❌

**问题代码：**
```csharp
// 3. 将结果呈现在界面上
RenderPanel.Text = htmlResult;  // ❌ Avalonia 没有这种用法
```

**实际情况：**
- Avalonia 是原生 UI 框架，不能直接将 HTML 赋值给 Text 属性
- 需要：(1) 使用 WebView 控件嵌入 HTML，或 (2) 将 Markdown 转换为 Avalonia 原生控件树
- 原文声称的"彻底摆脱浏览器内核"与后续伪代码矛盾

**正确方案：**
- 方案 A：使用 `Markdown.Avalonia` 组件（将 Markdown 转为原生控件）
- 方案 B：使用 `WebView` 控件（但仍依赖系统 WebView）
- 方案 C：自行实现 HTML→Avalonia 渲染层（工作量巨大）

### 3.2 分享功能缺乏关键设计 🔴

**缺失的安全与合规考虑：**

1. **数据隐私**
   - 未说明是否端到端加密
   - 未定义数据保留策略（上传后存多久？）
   - 未提及 GDPR/个人信息保护法合规

2. **访问控制**
   - 是否需要密码保护？
   - 是否支持链接过期？
   - 是否支持访问次数限制？

3. **内容审核**
   - 如何防止非法内容上传？
   - 是否需要敏感词过滤？

4. **成本控制**
   - 存储和带宽成本估算
   - 是否有免费额度限制？

**建议补充：**
```
分享功能安全规范：
├── 传输层：TLS 1.3 强制加密
├── 存储层：AES-256 加密，支持用户自定义密码
├── 访问层：
│   ├── 可选密码保护（6位数字/字母组合）
│   ├── 链接有效期（默认 7 天，最长 30 天）
│   └── 访问次数限制（默认无限，可设为 1-100 次）
├── 内容审核：
│   ├── 自动扫描敏感内容（调用第三方 API）
│   └── 举报机制（用户可标记违规内容）
└── 数据生命周期：
    ├── 默认 30 天自动删除
    ├── 用户可手动提前删除
    └── 定期清理过期数据
```

### 3.3 文件关联实现细节缺失

**问题：**
- Windows、macOS、Linux 的文件关联机制完全不同
- 未提及注册表操作、权限提升、卸载清理
- 未考虑多实例同时打开同一文件的场景

**建议补充：**
```
跨平台文件关联方案：

Windows:
- 注册表写入 HKCU（无需管理员权限）
- 使用 ProgID 机制而非直接关联扩展名
- 支持"打开方式"对话框

macOS:
- Info.plist 中声明 CFBundleDocumentTypes
- 需要代码签名才能正确关联

Linux:
- .desktop 文件 + xdg-mime 命令
- 支持 MIME 类型关联
```

## 四、缺失的关键功能模块

### 4.1 实时预览与文件监听

**核心场景：** 用户在编辑器中修改 MD 文件，SnapMD 应自动刷新预览

**建议方案：**
```rust
// Tauri 方案（使用 notify crate）
use notify::{Watcher, RecursiveMode, watcher};
use std::sync::mpsc::channel;
use std::time::Duration;

fn watch_file(path: &str, tx: std::sync::mpsc::Sender<()>) {
    let (fsw_tx, fsw_rx) = channel();
    let mut watcher = watcher(fsw_tx, Duration::from_millis(200)).unwrap();
    watcher.watch(path, RecursiveMode::NonRecursive).unwrap();

    loop {
        match fsw_rx.recv() {
            Ok(_) => {
                let _ = tx.send(());  // 通知前端刷新
            }
            Err(e) => eprintln!("Watch error: {}", e),
        }
    }
}
```

## 五、竞品分析与差异化策略

### 5.1 主要竞品

| 竞品 | 定位 | 优势 | 劣势 |
|-----|-----|-----|-----|
| **Typora** | 所见即所得编辑器 | 用户体验极佳，品牌认知度高 | 收费（$14.99），启动较慢 |
| **MarkText** | 开源 Typora 替代品 | 免费开源，跨平台 | 维护不活跃，bug 较多 |
| **VS Code + Markdown 插件** | 代码编辑器附属功能 | 功能强大，生态丰富 | 启动慢，资源占用高 |
| **Obsidian** | 知识管理工具 | 插件生态强大，双向链接 | 功能过重，非纯粹预览器 |
| **Markdown Viewer（浏览器扩展）** | 轻量预览 | 无需安装桌面应用 | 无法处理本地文件（需配置） |

### 5.2 SnapMD 差异化定位

**核心卖点（需强化）：**
1. **双击秒开**：比 Typora 快 3-5 倍的启动速度
2. **零配置**：无需任何设置，安装即用
3. **一键分享**：竞品普遍缺乏的功能，但需解决安全信任问题

**建议差异化方向：**
```
定位口号："你的 Markdown 秒开预览器，分享只需一个链接"

核心场景：
├── 场景 1：快速预览（替代"右键→打开方式→浏览器"）
├── 场景 2：团队协作文档分享（替代"发文件→对方打开"）
├── 场景 3：技术博客/文档快速发布（替代"部署到 GitHub Pages"）
└── 场景 4：移动端预览（替代"发送到手机"）
```

## 六、性能指标建议

### 6.1 关键性能目标（建议补充）

| 指标 | 目标值 | 测量方法 |
|-----|-------|---------|
| 冷启动时间 | < 500ms（1MB 文件） | 从双击到窗口显示 |
| 热启动时间 | < 100ms（已运行，切换文件） | 从命令行调用到渲染完成 |
| 内存占用（基础） | < 80MB（Tauri）/ < 50MB（Avalonia） | 空闲状态 |
| 内存占用（大文件） | < 200MB（10MB MD 文件） | 渲染完成状态 |
| 安装包体积 | < 20MB（Tauri）/ < 80MB（Avalonia 自包含） | 最终产物 |
| 文件打开速度 | < 200ms（1MB 文件渲染完成） | 从文件读取到 UI 更新 |

### 6.2 性能优化策略

**Tauri 方案：**
- 启用 WebView2 的"启动增强"（Startup Boost）
- 使用虚拟滚动处理大文件
- Markdown 解析使用 Web Worker 避免阻塞 UI

**Avalonia 方案：**
- 使用 Avalonia 11 的新 Composition Renderer
- Markdown 解析异步化（Task.Run）
- 图片懒加载 + 缓存策略

## 七、安全合规清单

### 7.1 桌面端安全

- [ ] 文件读取权限：仅访问用户显式打开的文件
- [ ] 网络权限：仅在分享功能激活时建立连接
- [ ] 代码签名：Windows（EV 证书）/ macOS（Developer ID）
- [ ] 自动更新：支持 HTTPS 验证的增量更新
- [ ] 沙箱隔离：WebView 进程与主进程分离

### 7.2 云端分享安全

- [ ] 传输加密：TLS 1.3 强制
- [ ] 存储加密：AES-256-GCM
- [ ] 用户密码：bcrypt/scrypt 哈希存储
- [ ] 速率限制：防止暴力破解分享链接
- [ ] 内容审核：集成第三方 API（如阿里云内容安全）
- [ ] 日志审计：记录所有分享/访问行为
- [ ] 数据删除：用户可随时删除，30 天自动清理

### 7.3 隐私合规

- [ ] 隐私政策：明确说明数据收集范围
- [ ] GDPR 合规：欧盟用户数据可迁移/删除
- [ ] 个人信息保护法：中国用户数据本地化存储
- [ ] 匿名选项：分享时可选择匿名/实名
